## 总体结论

结论非常明确：

1. `translate.js/translate.js` 语法层面没有损坏，`node --check` 可通过。
2. 但逻辑层面存在多处确定性 bug，不是“可能有问题”，而是已经可以确认的问题。
3. 当前实现本质上是一个 `Web DOM 自动翻译器`，并不是一个可直接跨微信小程序、原生 Android/iOS/鸿蒙复用的通用多语言 SDK。
4. 真正值得跨端复用的是“翻译内核”和“对象翻译能力”，不是 DOM 扫描、iframe、postMessage、MutationObserver 这一层。

## 子代理复核结论

## 已确认问题清单

以下按优先级排序。

### P0-1 `config.get()` 导出 `nomenclature` 数据错误

位置：

- `translate.js/translate.js:182`

问题：

- `data.nomenclature = translate.images.queues;`
- 这里明显应当导出 `translate.nomenclature.data`，但实际导出了图片替换队列。

影响：

- 配置导出结果错误。
- iframe 自动注入时通过 `config.get()/set()` 复制配置，会把术语库弄丢。
- 用户导出配置后再导入，得到的数据不是原始状态。

结论：

- 这是确定性 bug。

---

### P0-2 `config.set()` 与注释不一致，传部分配置会崩溃

位置：

- `translate.js/translate.js:222`
- `translate.js/translate.js:227`
- `translate.js/translate.js:248`
- `translate.js/translate.js:266`
- `translate.js/translate.js:301`

问题：

- 注释写的是“不想设置的项可以不传入”。
- 但实现中直接访问 `data.language.select.show`、`data.request.api.host`、`data.ignore.text` 等深层字段。
- 只要上层对象不存在，就会直接抛 `TypeError`。

最小复现：

```js
translate.config.set({});
```

会触发类似错误：

```js
TypeError: Cannot read properties of undefined (reading 'select')
```

影响：

- 配置导入 API 不可安全使用。
- iframe 配置同步也会受影响。
- 后续扩展配置项越多，出错概率越高。

结论：

- 这是确定性 bug。

### P0-3 `config.set()` 错把 `whole.enableAll` 写到 `listener.use`

位置：

- `translate.js/translate.js:284`

问题：

- 代码是：

```js
translate.listener.use = data.whole.enableAll;
```

- 正确目标应当是 `translate.whole.isEnableAll`。

影响：

- 导入配置时会把“整体翻译开关”错误地恢复成“动态监听开关”。
- 行为和导出配置完全不对称。

结论：

- 这是确定性 bug。

---

### P0-4 `config.set()` 会破坏 `translate.ignore.class.push()` 能力

位置：

- `translate.js/translate.js:275`
- `translate.js/translate.js:1411`

问题：

- `config.set()` 里直接执行：

```js
translate.ignore.class = data.ignore.class;
```

- 但运行态的 `translate.ignore.class` 是一个对象，不只是数据容器，它还有 `push()` 方法。

影响：

- 一旦配置导入覆盖掉整个对象，后续 `translate.ignore.class.push(...)` 很可能失效。
- 属于“配置导入破坏运行 API”的问题。

结论：

- 这是确定性 bug。

影响：

### P1-1 `translate.request.translateText()` 对重复文本回填错位

位置：

- `translate.js/translate.js:9735-9805`

问题：

- 代码用 `hash -> 单个 index` 来记原始位置。
- 如果同一批请求里有重复文本，后一个索引会覆盖前一个索引。

最小复现思路：

```js
texts = ['hello', 'hello']
```

当前逻辑最终只会把结果回填到最后一个位置。

影响：

- 对重复文案的翻译结果会丢失或错位。
- 对对象翻译、批量翻译、资源预翻译都属于高风险问题。

结论：

- 这是确定性 bug。

### P1-2 `translate.request.translateText()` 错误响应后只记日志不返回

位置：

- `translate.js/translate.js:9783`

问题：

- `responseData.result != 1` 时只是打印日志，没有 `return`。
- 后续仍然继续访问 `responseData.text.length`。

影响：

- 错误响应会继续进入正常处理流。
- 可能触发二次异常，掩盖真实错误原因。

结论：

- 这是确定性 bug。

### P1-3 `client.edge.translate()` 大文本分片回包靠长度猜分片位置

位置：

- `translate.js/translate.js:8904-8992`
- 关键点：`translate.js/translate.js:8961`

问题：

- 多个分片异步返回时，通过“哪个分片长度和当前返回长度一样”来判断分片位置。
- 如果存在两个同长度分片，就可能匹配到错误的分片。

影响：

- 大文本翻译结果错位。
- 异步乱序时风险更大。

结论：

- 这是确定性高风险 bug。

### P1-4 `execute()` 排队执行对单节点参数处理不安全

位置：

- `translate.js/translate.js:3431-3437`

问题：

- 当 `translate.state != 0` 时，会进入 waitingExecute 分支。
- 这里直接对 `docs` 调用 `slice()`。
- 但 `execute()` 实际支持传单个节点，不一定总是数组。

影响：

- 在翻译过程中再次调用 `translate.execute(node)` 时，可能直接抛错。

结论：

- 这是确定性 bug。

---

### P1-5 `waitingExecute.add()` 每次入队都新开一个 `setInterval`

位置：

- `translate.js/translate.js:2984-2996`

问题：

- 每入队一次就创建一个新的轮询器。
- 多个轮询器会同时观察同一个 `translate.state` 和同一个队列。

影响：

- 可能导致重复触发、空取队列、时序竞争。
- 这也是代码里已有“queue 已空但还往外取”日志的根因之一。

结论：

- 这是确定性设计缺陷。

---

### P1-6 MutationObserver 批处理重复节点时会漏翻

位置：

- `translate.js/translate.js:2610-2621`

问题：

- 命中重复节点后用了 `break`。
- 这里应当是跳过当前节点，继续处理后面的节点，因此应该用 `continue`。

影响：

- 同一批新增节点中，只要前面出现一个重复节点，后面的唯一新节点也可能被跳过。

结论：

- 这是确定性 bug。

## 具体修复方案

### 方案 A：重做 `config.get()/set()`

建议：

1. 以 `new translate.config.data()` 生成默认结构。
2. 对用户传入配置做深合并，而不是直接深层读取。
3. 统一维护一份“配置字段映射表”，避免手写一堆重复赋值。
4. `ignore.class` 只回填 `data` 和 `conditionFunction`，不能覆盖对象本身。
5. 对 `request.api.language` 同时支持字符串和对象数组。

预期收益：

- 导出导入可逆。
- iframe 配置同步稳定。
- API 更符合文档。

### 方案 B：重写批量翻译结果回填逻辑

建议：

1. 不再使用 `hash -> 单个 index`。
2. 改成：

```js
pending = [
  { index: 0, text: 'hello', hash: '...' },
  { index: 1, text: 'hello', hash: '...' }
]
```

3. 网络返回第 `i` 条结果时，直接回填 `pending[i].index`。
4. 如果需要缓存去重，也应使用 `hash -> indices[]`。

预期收益：

- 重复文本不会丢失。
- 对 `transObject()`、资源预翻译都更可靠。

### 方案 C：给 edge 分片增加明确 `chunkIndex`

建议：

1. 分片前记录每个 chunk 的顺序号。
2. 每个异步回调都只合并到自己的 `chunkIndex`。
3. 最终按 `chunkIndex` 排序拼回完整结果。

预期收益：

- 摆脱“按长度猜分片位置”。
- 异步乱序下依然稳定。

### 方案 D：重构 waitingExecute 为单调度器

建议：

1. 队列项统一结构化：

```js
{
  docs: normalizedDocs,
  to: translate.to,
  triggerId: '...'
}
```

2. 维护唯一一个调度器。
3. `execute()` 一开始就将 `docs` 规范成数组。
4. 调度器一次只消费一个任务。

预期收益：

- 消除并发轮询竞争。
- 避免对单节点 `.slice()` 崩溃。

### 方案 E：运行时适配层

建议：

1. 抽象 `storageAdapter`。
2. 默认优先：
   - `localStorage`
   - 内存 fallback
3. 初始化失败必须 `translate.log(e)`。

预期收益：

- WebView、沙箱、隐私模式可用性更高。

## 后续扩展方向

最值得继续扩展的方向有四个。

### 1. 抽 `translate-core`

把这些纯能力抽出去：

- 语言状态
- 请求层
- 缓存层
- 自定义术语
- `translate.request.translateText()`
- `translate.js.transString()`
- `translate.js.transObject()`

### 2. 抽 `web-adapter`

把这些 Web 专属能力单独隔离：

- DOM 扫描
- MutationObserver
- iframe 注入
- postMessage
- select UI
- cookie / localStorage

### 3. 增加资源预翻译工具

把对象翻译能力做成 CLI：

- 输入配置对象/菜单对象/表单 schema
- 输出多语言 JSON

### 4. 增强调试工具

至少补齐：

- 为什么某段文本没翻
- 命中了哪条 ignore 规则
- 当前翻译任务状态
- 当前语言状态机
- 当前请求失败原因

## 跨微信小程序、Android、iOS、鸿蒙的方案

## 先说结论

当前实现不能原样搬过去。

原因：

1. 当前核心依赖浏览器 DOM。
2. 当前依赖 `MutationObserver`、iframe、postMessage。
3. `uniapp` 现有方案本质也是 H5/renderjs 方案。
4. 仓库现有文档已经明确说明不支持微信小程序。

相关文件：

- `extend/uniapp/README.md`
- `extend/uniapp/组件/translate.vue`
- `extend/uniapp/组件/switch-language.vue`

### 现有 uniapp 方案的额外问题

这两个组件的判断逻辑也不严谨：

- `extend/uniapp/组件/translate.vue`
- `extend/uniapp/组件/switch-language.vue`

都用了：

```js
typeof window.translate === 'function'
```

但 `translate` 实际是对象，不是函数。

这会造成“已加载但误判未加载”的问题。

## 正确的跨端架构建议

### 架构拆分

建议拆成三层：

1. `translate-core`
2. `translate-web-adapter`
3. `translate-platform-adapter`

### `translate-core` 负责什么

- 当前语言状态
- 翻译请求
- 缓存
- 术语库
- `transObject/transString`
- 离线语言包

这层禁止直接依赖 DOM。

### `translate-web-adapter` 负责什么

- DOM 扫描
- MutationObserver
- iframe 联动
- select UI
- 浏览器存储

用于：

- H5
- WebView 页面
- uniapp 的 H5/App-H5 页面

### `translate-platform-adapter` 负责什么

用于：

- 微信小程序
- Android 原生
- iOS 原生
- 鸿蒙原生

职责：

- 读取/保存当前语言
- 触发 UI 状态刷新
- 加载本地资源包
- 对动态数据调用 `translate-core`

## 静态文案和动态文案要分开

建议明确分层：

### 静态 UI 文案

如：

- 按钮
- 标题
- 菜单
- 表单标签

必须走：

- `t(key)` + 本地资源包

### 动态业务文本

如：

- 文章正文
- 评论
- 商品描述
- 后台返回的配置对象

才走：

- `translate.request.translateText()`
- `translate.js.transObject()`

这样跨端才稳。

## 各端落地建议

### 微信小程序

方案：

- 使用 `localeStore + t(key)`。
- 语言切换时统一刷新 `setData`。
- 动态文本调用 `translate-core`。
- 不要尝试 DOM 扫描。

### Android

方案：

- 静态文案：资源包或 JSON。
- 动态文本：`translate-core`。
- 语言状态：`DataStore/MMKV/ViewModel/StateFlow`。
- 如果页面本身是 WebView，可直接继续复用 `web-adapter`。

### iOS

方案：

- 静态文案：`String Catalog` 或 `Localizable.strings`。
- 动态文本：`translate-core`。
- 语言状态：`UserDefaults + NotificationCenter/ObservableObject`。

### 鸿蒙

方案：

- 静态文案：`ResourceManager` / `string.json`。
- 动态文本：`translate-core`。
- 语言状态：`Preferences/AppStorage`。

## 推荐迁移顺序

建议按以下顺序推进：

1. 先修当前 Web 端 P0/P1 bug。
2. 抽出 `translate-core`。
3. 统一多语言资源格式，如 `locales/{lang}/{namespace}.json`。
4. 保留 `web-adapter` 兼容 H5。
5. 新做微信小程序 adapter。
6. 再做 Android/iOS/鸿蒙 adapter。

## 一句话总结

### 当前剩余待处理问题

以下问题仍未处理，留待下一轮继续：

1. `translate.execute()` 在 busy 分支里对 `docs` 直接 `slice()`，当外部传单个 DOM 节点且当前处于忙碌状态时，会直接抛错，任务无法入队
2. `execute.trigger_Trigger` 触发位置早于排队判定，可能造成“排队任务先拿到临时 uuid、先触发一次 hook，真正出队时再执行一次”的双重语义问题
3. `config.set()` 直接覆盖 `translate.ignore.class` 整个对象，会破坏其 `push()` 方法
4. `config.get()` 导出了 `request.api.translate`，但 `config.set()` 仍未回填该字段
5. `selectLanguageTag.render()` 在语言列表异步加载失败时，会卡死在 `alreadyRender = true` 状态，后续无法自动重试
6. `MutationObserver` 批处理重复节点时使用 `break` 而不是 `continue`，可能导致后续唯一新节点漏翻
7. `localStorage` 直接使用且无降级兜底，在受限 iframe、某些 WebView、隐私模式等场景存在运行风险
8. 初始化 `translate.init()` 的异常仍被静默吞掉，不利于排障
9. `selectionTranslate` 仍把坐标状态放在全局变量中，存在状态泄漏风险
10. `postMessage` 默认来源边界过宽，属于设计层面的安全收敛问题
